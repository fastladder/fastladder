# 脱Docker、脱複雑性。FastladderをGo言語でリメイクして『HASHIGO』を作った話

かつて「Fastladder」という伝説的なRSSリーダーがありました。高速で、キーボードショートカットでサクサク消化できるその体験は、多くのエンジニアを魅了しました。しかし時が経ち、オリジナルのRails製コードベースを動かすにはDockerコンテナのオーケストレーションが必要になり、Rubyのバージョン管理に追われ、ローカルでサッと動かすには少しばかり「重い」存在になってしまいました。

「もっとシンプルでいいんじゃないか？」

ふとそう思いました。Dockerもいらない。Rubyのセットアップもいらない。シングルバイナリひとつあれば、どこでも動く。データベースはSQLiteで十分。そんな「原点回帰」とも言えるRSSリーダーが欲しくなったのです。

そうして生まれたのが、今回開発した**『HASHIGO（ハシゴ）』**です。

## なぜ今、RSSリーダーを自作するのか？

SNSのタイムラインはアルゴリズムに支配されています。便利ですが、そこには「自分が選んだ情報」という感覚が希薄です。RSSは違います。自分が興味を持ったサイトの更新情報を、漏らさず、時系列順に受け取る。この「情報の主体性」を取り戻すためのツールとして、RSSリーダーは今なお最強のツールです。

しかし、既存のWebサービス型RSSリーダー（Feedlyなど）は便利ですが、オフラインでは見られなかったり、UIの変更に振り回されたりします。かといって、セルフホスティング型のリーダーは構築が面倒です。

「ローカルで完結し、自分の好きなエディタやターミナルで操作できる、自分だけの情報基地が欲しい」

この欲求を満たすには、自分で作るしかありませんでした。

## コンセプト：少ない言語、少ないコンポーネント

HASHIGOの開発にあたって掲げた哲学はシンプルです。

1.  **脱Docker**: コンテナ管理の煩わしさから解放されること。
2.  **Single Binary**: `.exe` (またはLinuxバイナリ) ひとつで動作すること。
3.  **Local First**: データは全てローカルのSQLiteファイルに保存すること。
4.  **CLI & TUI**: マウスを使わず、キーボードだけで完結すること。

選んだ言語は **Go** です。
Goはクロスコンパイルが容易で、シングルバイナリの生成に最適です。そして何より、標準ライブラリが強力で、外部依存を最小限に抑えられます。

## 技術スタック：Go + Bubbletea + SQLite

HASHIGOの構成要素は驚くほどシンプルです。

*   **言語**: Go (1.25)
*   **TUIフレームワーク**: Bubbletea
*   **RSSパーサー**: gofeed
*   **データベース**: modernc.org/sqlite (CGO不要のPure Go実装)

これだけです。
特にこだわったのが「TUI (Text User Interface)」の実装です。Go製のTUIフレームワークである `Bubbletea` を採用し、かつてコマンドラインツール界隈で一世を風靡した `peco` のような操作感を目指しました。

ターミナルで `hashigo` と打つだけで、未読記事がリストアップされます。
`/` キーを押せばインクリメンタルサーチモードになり、タイトルやフィード名で瞬時に絞り込みができます。
気になる記事があれば `Enter`。すると既定のブラウザで記事が開き、同時にバックグラウンドで「既読」ステータスに更新されます。

この「ターミナルから一歩も出ずに情報を捌く」体験は、一度味わうと病みつきになります。Webブラウザのタブを行ったり来たりする必要はありません。

## もうひとつの未来：Bun + Hono + Svelte

実は、HASHIGOプロジェクトにはもうひとつの「未来」も用意されています。
「やっぱりターミナルだけじゃ画像が見えないし、スマホからも見たいよね」というニーズに応えるための、Web UIプランです。

ここでも「シンプルさ」へのこだわりは捨てていません。
採用したのは **Bun + Hono + Svelte** という、現代における最速・最軽量の組み合わせです。

*   **Runtime**: Bun (Node.js互換かつ爆速)
*   **Backend**: Hono (Web標準に準拠した軽量フレームワーク)
*   **Frontend**: Svelte (仮想DOMを持たないコンパイラ)

これらを組み合わせることで、TypeScript一本でフロントエンドからバックエンドまでを記述しつつ、非常に高速な動作を実現します。
もちろん、データベースはGo版と同じ `hashigo.db` (SQLite) を共有します。つまり、普段はターミナルでサクサク読み、休日はリッチなWeb UIで画像付きの記事を楽しむ……そんな使い分けが可能になる予定です。

## 今後の展望：HTMXとChrome拡張

さらに、開発ロードマップには以下の機能を予定しています。

1.  **YAMLによるフィード管理**:
    `feeds.yaml` というファイルを置いておけば、起動時に勝手に同期してくれる機能。「設定＝コード」としてGitで管理できるようになります。RSSリーダーの移行も、このファイルをコピペするだけです。

2.  **Chrome拡張機能**:
    「あ、このブログ面白い」と思った瞬間に、ブラウザのボタンをポチッ。それだけでローカルのHASHIGOにフィードが登録される。そんなシンプルな連携機能を実装予定です。

3.  **HTMXによるWeb UI**:
    Svelteも素晴らしいですが、もっとストイックに「GoのバイナリにHTMLテンプレートを埋め込む」形でのWeb UIも模索しています。HTMXを使えば、Reactなどの重厚なJSフレームワークを使わずに、SPAのような快適な操作感を実現できます。これも「Single Binary」の哲学に合致します。

## おわりに：道具を自作する喜び

エンジニアにとって、自分が毎日使う道具を自作し、磨き上げていくことほど楽しいことはありません。
HASHIGOはまだ生まれたばかりのツールですが、すでに私の情報収集フローには欠かせない存在になりつつあります。

「機能が足りないなら足せばいい。遅いなら直せばいい」

この全能感こそが、個人開発の醍醐味です。
もしあなたも、既存のRSSリーダーに何かしらの不満を抱いているなら、ぜひHASHIGOを触ってみてください。あるいは、フォークして自分だけの「最強のハシゴ」を作ってみるのもいいでしょう。

情報は、もっと自由に、もっとシンプルに扱えるはずです。
HASHIGOが、あなたの情報収集ライフを少しでも快適にできることを願っています。
